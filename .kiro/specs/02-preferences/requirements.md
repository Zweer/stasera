# Preferences Requirements

## Goal

Build a user taste profile through pairwise comparisons during onboarding,
and allow progressive refinement over time. The profile drives personalized
event recommendations.

## Core Flow

```
Onboarding (new user)
  → 8-10 pairwise comparisons ("Which do you prefer?")
  → User picks one + selects a reason (LLM-generated options)
  → Profile computed from choices
  → Profile visible to user, editable anytime

Refinement (ongoing)
  → Feedback on recommendations updates profile
  → User can trigger more comparisons anytime
```

## Features

### 1. Onboarding Quiz
- Present two options side by side
- User taps their preference
- Show 3-4 reason options (generated by LLM based on the pair)
- User selects why they preferred that option
- 8-10 rounds for initial profile

### 2. Comparison Options
- Initially: archetypal events (generic scenarios representing categories)
  - e.g., "Jazz in an intimate club" vs "Rock concert in a piazza"
- When DB has enough real events (>15): use actual upcoming events

### 3. Reason Generation
- LLM generates contextual "why" options for each pair
- e.g., "Because it's outdoors", "Because I like live music", "Because it's intimate"
- Reasons map to preference tags

### 4. Profile Computation
- Preference vector: dictionary of tag weights
  - e.g., `{"jazz": 0.8, "outdoor": 0.6, "romantic": 0.4, "high-energy": -0.2}`
- Updated incrementally with each comparison and feedback
- Weights decay over time (recent choices matter more)

### 5. Profile View
- User sees their taste profile as readable tags with strengths
- Can trigger additional comparisons to refine
- Clear explanation: "You seem to like X because of Y"

### 6. Archetype Library
- 8-10 pre-defined event archetypes covering Genova's nightlife spectrum
- Used for onboarding when real events are insufficient
- Each archetype has: title, description, genre, vibe, energy, setting

## Success Criteria

- New user completes onboarding in under 2 minutes
- Profile accurately reflects stated preferences (manual verification)
- Reason options feel natural and relevant to the pair
- Profile updates correctly after each comparison
- User can view and understand their profile

## Non-Goals

- Implicit tracking (location, time spent, etc.)
- Social comparison ("friends who like similar things")
- Automatic profile migration from other platforms
